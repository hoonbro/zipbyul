## 0. 전제 · 목표

백엔드 기획안 8-1의 **"단일 VM + Docker Compose"** 모델을 그대로 잇는다. 여기에 **리버스 프록시(Caddy)** 한 대를 앞에 두어:

- **단일 도메인**으로 web(PWA) + api를 함께 서빙 → **CORS 불필요**
- **자동 HTTPS**(Let's Encrypt) → 서비스워커·웹푸시·"홈 화면에 추가" 요건 충족
- `/internal/*` 외부 차단

> 빠른 출시·CDN이 우선이면 정적 호스팅 분리(Plan B, 10장)도 가능하나, 현 인프라 일관성을 위해 **A를 채택**한다.

------

## 1. 아키텍처 / 토폴로지

```
                  인터넷 (HTTPS :443)
                        │
                ┌───────▼────────┐   자동 TLS(Let's Encrypt)
                │     Caddy       │   단일 도메인 (예: app.zipbyul.kr)
                │  리버스 프록시   │
                └───┬─────┬───┬───┘
        /v1/* ┌─────┘     │   └─────┐ /*
              ▼           ▼         ▼
        ┌──────────┐  /internal/*  ┌──────────────┐
        │ api:8080 │   → 403 차단   │ web 정적(dist)│  SPA fallback
        └────┬─────┘               └──────────────┘
             │
   ┌─────────┼─────────┐
   ▼         ▼         ▼
postgres   redis   (flyway one-shot)
   ▲
   └── collector (수집·정규화, 쓰기)
```

- **한 도메인**에서 `/v1`·`/internal`은 api로, 나머지는 web 정적으로 라우팅.
- web과 api가 **같은 origin** → 브라우저 CORS 자체가 발생하지 않음.

------

## 2. 컴포넌트별 배포

| 컴포넌트 | 방식 | 비고 |
| ---- | ---- | ---- |
| **caddy** (신규) | Compose 서비스. 정적 서빙 + 리버스 프록시 + 자동 TLS | 443/80 노출. Caddyfile 1개 |
| **web** (신규) | multi-stage Dockerfile: `pnpm build` → `dist/` | 빌드 산출물을 Caddy가 서빙(공유 볼륨) 또는 web 이미지가 dist만 제공 |
| api | 기존 Compose(`build: ./api`) | 외부 포트 미노출(Caddy 경유만) 권장 |
| collector | 기존(`--schedule`) | 외부 노출 없음 |
| postgres / redis / flyway | 기존 | postgres 포트도 외부 미노출 권장 |

> **정적 서빙 위치**: web Dockerfile은 `dist/`만 만들고, Caddy가 그 볼륨을 `file_server`로 서빙하는 구성이 단순(프록시+정적이 Caddy 한 곳). web을 자체 nginx로 띄워 Caddy가 프록시해도 무방.

------

## 3. 라우팅 규칙 (Caddyfile 개요)

```
app.zipbyul.kr {
    encode zstd gzip

    # 내부 전용 — 외부 차단
    handle /internal/* {
        respond 403
    }
    # API
    handle /v1/* {
        reverse_proxy api:8080
    }
    # 외부 헬스체크/업타임 모니터링용 (그 외 actuator는 미노출)
    handle /actuator/health {
        reverse_proxy api:8080
    }
    # PWA 정적 + SPA fallback
    handle {
        root * /srv/web
        # 해시 에셋: 장기 불변 캐시
        @assets path /assets/*
        header @assets Cache-Control "public, max-age=31536000, immutable"
        # 진입 HTML·서비스워커: 항상 최신(고착 방지)
        @nocache path /index.html /sw.js /firebase-messaging-sw.js /manifest.webmanifest
        header @nocache Cache-Control "no-cache"
        try_files {path} /index.html
        file_server
    }
}
```

- `try_files … /index.html` 로 클라이언트 라우팅(새로고침) 대응. 실제 파일(SW·manifest)은 fallback 전에 파일로 서빙됨.
- **캐시 전략(PWA 필수)**: 해시된 `/assets/*`는 immutable 장기 캐시, `index.html`·`sw.js`·`firebase-messaging-sw.js`·`manifest`는 `no-cache` → 신버전 배포 시 사용자 고착 방지.
- `/internal/*`는 403(또는 사내 IP allowlist). api의 `INTERNAL_API_TOKEN` 가드와 이중 방어.
- `/actuator/health`만 프록시(외부 모니터링용). 그 외 actuator 엔드포인트는 노출 안 함.

## 4. 환경변수 / 시크릿

### 빌드 타임 (web) — Vite가 번들에 인라인

| 변수 | prod 값 | 메모 |
| ---- | ---- | ---- |
| `VITE_API_BASE_URL` | **빈 문자열 `''`** | 같은 origin → `/v1/...` **상대 호출**(CORS 제거). api.ts가 `?? 'localhost:8080'`이라 빈값이면 상대경로 |
| `VITE_VAPID_PUBLIC_KEY` | 백엔드와 동일 공개키 | |
| `VITE_FIREBASE_API_KEY` / `_MESSAGING_SENDER_ID` / `_APP_ID` | Firebase 콘솔 웹 앱 config | 공개값(안전) |
| `public/firebase-messaging-sw.js` | 위 config 실값 치환 | **SW엔 env 주입 안 됨 → 빌드 시 치환/생성 필요** |

### 런타임 (api/collector) — `.env`(미커밋)

`POSTGRES_*`, `DATA_GO_KR_SERVICE_KEY`, `ECOS_API_KEY`, `FCM_PROJECT_ID`, `FCM_CREDENTIALS_JSON=/run/secrets/firebase-service-account.json`, `WEBPUSH_VAPID_*`, `INTERNAL_API_TOKEN`(필수), `NOTIFICATION_CONSUMER_ENABLED=true`.

- `CORS_ALLOWED_ORIGINS`: 같은 origin이라 **실질 불필요**(설정 무방).
- 시크릿: `.env` + `secrets/`(ro 볼륨), 저장소 커밋 금지(.gitignore 적용 확인됨).

> **강점 — web 이미지 환경 비종속**: `VITE_API_BASE_URL=''`(상대경로)라 API URL이 빌드에 안 박힘 → **staging/prod가 같은 web 이미지 재사용** 가능. 환경별로 다른 값은 Firebase/VAPID(프로젝트 단위)뿐.

## 5. 릴리스 순서 (백엔드 기획안 8-3 준수)

```
1) 이미지 빌드        web / api / collector
2) flyway migrate     스키마 먼저 적용(단일 단계)
3) 앱 롤아웃          api · collector 교체
4) 정적 교체 + caddy reload   web dist 갱신
```

- 마이그레이션 → 앱 순서 고정(8-3). MVP는 api 교체 시 짧은 순단 허용(무중단은 v1 이후).
- `pgdata` 볼륨 백업 정책은 별도(9장 TODO).

## 6. CI/CD 방향

- **현재**: GitHub Actions 경로필터로 `web/api/collector/db` 각각 build·lint·test (**CD 없음**).
- **추가(제안)**:
  - main 머지 → 이미지 빌드 후 레지스트리(GHCR) 푸시
  - VM에서 `docker compose pull && (flyway migrate) && up -d`
  - web의 `VITE_*`·SW config는 **빌드 스텝에서 GitHub Actions Secrets로 주입**
- db 잡은 이미 flyway migrate로 마이그레이션 적용 검증 중.

## 8. 보안

| 항목 | 조치 |
| ---- | ---- |
| **prod 포트 노출** | **Caddy만 80/443 노출.** prod compose override에서 `api`(8080)·`postgres`(5432)·`redis`(6379) **포트 매핑 제거** — 안 그러면 `:8080/internal`로 Caddy 403 우회됨 |

| DB/Redis | 외부 포트 미노출(Compose 내부 네트워크만) |
| 전송 | 전구간 HTTPS(Caddy), HSTS 기본 |
| CSP | (TODO) SW가 `gstatic.com`에서 firebase importScripts → CSP 도입 시 허용 도메인 명시 |

## 10. 대안 — Plan B (참고)

정적 CDN(Vercel / Cloudflare Pages / Netlify)에 web, api는 VM + `CORS_ALLOWED_ORIGINS`에 웹 도메인 등록.

- 장점: 웹 배포 DX·CDN. 단점: **두 origin + CORS 관리**, api도 공개 HTTPS 도메인 필요.
- 단일 호스트 일관성을 위해 **본 기획은 A 채택**. 트래픽 증가 시 web만 B로 분리 가능.
