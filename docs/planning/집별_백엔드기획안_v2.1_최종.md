## 2. 전체 시스템 아키텍처

집별은 본질적으로 **"공식 공공 API를 주기적으로 수집·정규화하고, 변화를 감지해 개인화 피드·알림으로 전달하는 데이터 파이프라인 + 비회원 개인화 서빙 서비스"**다. 실시간 트랜잭션이 적고 배치성 수집과 이벤트성 알림이 중심이다.

```
┌──────────────────────────────────────────────────────────────┐
│            외부 데이터 소스 (MVP: 공식 API)                     │
│  청약홈 │ LH │ 국토부 실거래 │ ECOS(서울 CSI·금리) │ 서울데이터   │
└───────┬──────────────────────────────────────────────────────┘
        │ API 폴링
╔═══════▼═══════════════════════════════╗  ← Python 영역 (쓰기)
║ 1. 수집(Collector)  raw_payload 저장    ║  스케줄러 배치
║ 2. 정규화(Normalizer) 표준스키마/멱등upsert║  dedup_hash + first_seen_at
║ 3. 분석(Enricher)  base_score 산정/이벤트 ║  domain_event(outbox) 발행
╚═══════╤═══════════════════════════════╝
        │ INSERT/UPSERT + domain_event(status=NEW)
┌───────▼───────────────────────────────┐
│        공유 DB (PostgreSQL · SSOT)       │  Flyway 단일 관리
│  정규화 테이블 · market_index · outbox   │
└───────┬───────────────────────────────┘
        │ SELECT (읽기) / outbox 폴링
╔═══════▼═══════════════════════════════╗  ← Spring 영역 (읽기·서빙)
║ 4. API 서버   비회원 개인화 합성          ║
║ 5. 알림 계층   사용자별 점수 계산/룰/FCM   ║  outbox consumer + fan-out
╚═══════╤═══════════════════════════════╝
        │
┌───────▼─────────┐
│   프론트(PWA)     │
└─────────────────┘
```

계층 분리 이유: **수집 실패가 서빙 장애로 번지지 않게** 하고, 소스가 늘 때 Adapter만 추가하면 되게 하기 위해서. **쓰기는 Python, 읽기는 Spring**으로 방향을 고정해 두 스택이 서로를 직접 호출하지 않는다(공유 DB가 유일한 계약).

## 3. 데이터 수집 계층 설계

### 3-1. 역할 분담

| 계층 | 스택 | 담당 |
| ---- | ---- | ---- |
| 수집·정규화 | **Python** | 공공 API 호출, 표준 스키마 변환, 중복제거, 멱등 적재(`raw_payload`→정규화 테이블) |
| 분석·가공 | **Python** | base_score 산정, 변화 감지 → `domain_event` 발행 |
| 서빙 API | **Spring** | 비회원 홈 피드 합성, 캘린더/공고/실거래/집값전망 조회, 설정 저장 |
| 알림·인증 | **Spring** | outbox 소비 + 사용자별 점수 계산 + 룰 엔진 + FCM/WebPush, 익명 식별자 |

**쓰기/읽기 소유권 경계 (가장 중요한 약속)**: 정규화 테이블·`domain_event`는 Python이 쓰고 Spring은 읽기 전용. `anonymous_users`/`user_*`/`notification_logs`는 Spring이 쓰고 읽는다. 스키마는 Flyway 한 곳에서만 관리한다.

### 3-2. 소스별 수집 전략

| 소스 | 방식 | 주기 | 멱등 키(dedup_hash 원천) | 변화 감지 |
| ---- | ---- | ---- | ---- | ---- |
| 국토부 실거래 | API 폴링(자치구 × 최근 3개월) | 1일 1~2회 | 법정동+단지+전용면적+계약일+층+금액 해시 | **신규 행 = first_seen_at 부여 → `TRANSACTION_NEW`** |
| 청약홈 분양 | API 폴링 | 1일 2회 | 공고번호(pblancNo) | 신규 공고 / 접수시작 D-day / 마감 임박 |
| LH 공고 | API 폴링 | 1일 2회 | 공고ID(+단지) | 신규 공고 / 마감 임박 |
| ECOS 서울 CSI·금리 | API 폴링 | 1일 1회(지표 월 1회 갱신) | 통계코드+기준월 | 금리 변경 / CSI 급변 |
| 기준금리 결정일정 | **seed 등록** | 연 1회 | 회의일자 | 일정 D-day |

**실거래 보정 원칙**: 신고지연으로 최근 1~2주가 불완전하므로 **매번 최근 3개월치를 재조회**해 누락분을 보강한다. 재조회 시 이미 있는 행은 dedup_hash로 upsert(변화 없음), 없던 행만 INSERT 되며 그때 `first_seen_at = now()`가 곧 "신규 등록 시각"이다.

### 3-3. 신규 등록 감지 — `first_seen_at` (★ 핵심 변경)

국토부 실거래 API는 **계약일·금액·면적·층·법정동·단지명은 주지만 "신고 등록 시각"은 제공하지 않을 수 있다**(PoC에서 확인). 따라서 "신규 등록"은 외부 등록일이 아니라 **우리 파이프라인이 처음 본 시각**으로 정의한다.

- `real_estate_transactions.first_seen_at` = 해당 dedup_hash가 우리 DB에 **처음 INSERT 된 시각** (NOT NULL, default now()).
- upsert 결과가 INSERT(=신규)면 `TRANSACTION_NEW` 이벤트를 발행. UPDATE(=재조회 보정)면 발행 안 함.
- 사용자에게는 `first_seen_at`을 "신규 등록일"로 노출하되 "신고 지연으로 최근 데이터 일부 누락 가능" 고지를 동반한다.
- (만약 PoC에서 API가 진짜 등록일을 준다면 그 값을 `registered_at`에 추가로 저장하고 표시에 우선 사용한다. 감지 로직은 그래도 first_seen_at diff로 유지한다.)

### 3-4. 수집 파이프라인 4단계 (모든 Adapter 공통)

1. **fetch** — 원천 응답을 `raw_payload`에 그대로 저장(감사·재처리). 페이지네이션·지수백오프, 트래픽 한도(개발계정 보통 1만/일) 고려.
2. **normalize** — 지역(자치구/법정동코드), 날짜, 공급유형, 카테고리, 출처, 원문 링크를 표준 스키마로 매핑. `region_code` 매핑 테이블 사용.
3. **dedup** — 멱등 키 해시로 upsert(`dedup_hash` UNIQUE). 같은 공고 재수집은 변경분만 갱신.
4. **emit** — 신규/변경 감지 시 `domain_event(status=NEW, base_score=...)` 발행.

### 3-5. 호출량 산정 (개발계정 1만/일 한도 점검)

실거래는 콘텐츠를 전 자치구 미리 수집하는 게 합리적이므로 **서울 25개 구 전체** 기준으로 잡는다.

```
실거래 1회 수집 = 25개 구 × 주택유형 3(아파트/연립다세대/오피스텔)
                × 거래유형 2(매매/전월세 API 분리) × 최근 3개월
              = 25 × 3 × 2 × 3 = 450 호출
실거래 1일 2회 = 900 호출/일
청약홈/LH = 페이지네이션 포함 수십~수백 호출/일
ECOS = 월 1회 갱신, 1일 폴링 = 수 호출/일
─────────────────────────────────────────────
합계 ≈ 1,000~2,000 호출/일  →  개발계정 1만/일 한도 내 (여유 있음)
```

> 사용자 증가/소스 확장으로 한도 근접 시 운영계정 전환 신청 + 증분 수집·캐싱. 위 수치는 PoC에서 실제 페이지네이션 깊이로 재검증한다.

### 3-6. 안정성 (서비스 리스크 4 대응)

- **소스별 격리**: 잡 단위 분리. 한 소스 실패가 다른 소스·서빙에 영향 없음.
- **재시도/서킷브레이커**: 지수 백오프, 연속 실패 시 내부 알림.
- **수집 모니터링**: 잡별 마지막 성공시각·건수·실패율을 `collect_job_log`·`source_health_status`에 기록 → Grafana.
- **로컬/오프라인 개발**: PoC에서 뜬 실응답을 `fixtures/`에 저장해, 외부 API 없이도 normalize→serving을 개발·테스트한다(3-4의 fetch 단계만 mock).

## 6. 알림 정책과 중요도/별점 (★ 핵심 재설계)

핵심 원칙: **관심지역·관심유형과 맞는 고중요 이벤트만 즉시 발송, 나머지는 하루 요약**.

### 6-1. 중요도 점수 = base(이벤트 고유) + personalized(사용자별) (★)

중요도는 두 단계로 나뉜다. **하나의 이벤트라도 사용자마다 최종 점수가 다르기 때문**이다.

#### (1) base_score — 이벤트 고유 (Python이 산정해 `domain_event.base_score`에 저장)

| 조건 | 점수 |
| ---- | ---- |
| 신규 청약·공공임대 공고(`ANNOUNCEMENT_NEW`) | +2 |
| 접수 시작(`APPLICATION_START`) | +1 |
| 마감 D-3 이내 | +2 |
| 마감 D-1 또는 당일 | +3 |
| 기준금리 변경/결정(`RATE_DECISION`) | +2 |
| 집값전망 심리 급변(`MARKET_INDEX_UPDATED`) | +2 |
| 실거래 신규 등록(`TRANSACTION_NEW`) | +1 |
| 서울 전체 영향 정책 | +1 |

> base_score는 사용자와 무관하므로 이벤트당 1번만 계산·저장한다.

#### (2) personalized_delta — 사용자별 (Spring이 발송 대상 산정 시 계산, 저장 안 함)

| 조건 | 점수 |
| ---- | ---- |
| 관심지역 일치 | +2 |
| 관심유형 일치 | +2 |
| 관심지역·유형 모두 불일치 | -2 |

#### (3) 최종 점수·별점

```
final_score = clamp(base_score + personalized_delta, 0, 10)
stars       = convert(final_score)   # 아래 표
```

| final_score | 별점 | 알림 기본 판단 |
| ---- | ---- | ---- |
| 0~1 | ★☆☆☆☆ | 제외, 목록 하단 |
| 2~3 | ★★☆☆☆ | 하루 요약 중심 |
| 4~5 | ★★★☆☆ | 보통 이상 알림 |
| 6~7 | ★★★★☆ | 중요 알림 |
| 8~10 | ★★★★★ | 최우선 알림 |

> `IMPORTANT_ONLY`(별 3개↑) = `final_score >= 4` (목업 기준). "이미 마감" 이벤트는 점수가 아니라 **대상 필터 단계에서 제외**한다.

#### 산정 예시 (사용자: 마포구·청약 관심)

| 사례 | base | delta | final | 별점 |
| ---- | ---- | ---- | ---- | ---- |
| 마포구 LH 행복주택 공고 신규(행복주택 관심 켬) | +2 | +2(지역)+2(유형) | 6 | ★★★★☆ |
| 마포구 청약 마감 D-1 | +3 | +2+2 | 7 | ★★★★☆ |
| 기준금리 결정(정책·금리 관심 켬) | +2+1(서울전체) | +2(유형) | 5 | ★★★☆☆ |
| 마포구 실거래 신규 등록 | +1 | +2(지역) | 3 | ★★☆☆☆ |
| 강북구(비관심) 청약 신규 | +2 | -2 | 0 | ★☆☆☆☆ |

### 6-2. 알림 채널 / 강도

채널: **FCM·Web Push**(PWA) + 앱 내 알림함(`notification_logs`). 이메일·카카오 알림톡은 제외.

| 사용자 문구 | 서버 enum | 발송 기준 |
| ---- | ---- | ---- |
| 모든 알림 | `ALL` | 관심지역/유형 매칭 전체 즉시(뉴스 제외) |
| 중요 알림만 | `IMPORTANT_ONLY` | **final ≥ 4(별 3개↑)** 즉시 |
| 마감 임박만 | `DEADLINE_ONLY` | D-3/D-1/당일 마감만 즉시 |
| 관심지역만 | `REGION_ONLY` | 관심지역 매칭만. final ≥ 4 즉시, 그 외 하루 요약 |
| 하루 요약만 | `DAILY_DIGEST_ONLY` | 즉시 없음, 하루 1회 요약 |

> 목업 5종 기준. **"전체 끄기"는 alert_level이 아니라 푸시 마스터 토글(`user_devices.push_enabled=false` 또는 브라우저/FCM 권한 거부)**로 처리한다. v2.0의 `OFF`·`NORMAL_AND_ABOVE`는 제거됐다.

### 6-3. 이벤트 유형별 기본 발송 정책

| 이벤트 | base | 즉시 발송 조건 | 하루요약 |
| ---- | ---- | ---- | ---- |
| `ANNOUNCEMENT_NEW` | 2 | 관심지역/유형 매칭 + 강도 충족 | O |
| `APPLICATION_START` | 1(+D-day) | 관심지역/유형 매칭 | O |
| `APPLICATION_DEADLINE` | 2~3(D-day) | `DAILY_DIGEST_ONLY` 외 대부분 즉시 | O |
| `WINNER_ANNOUNCEMENT` | 0~2 | 저장/조회했거나 관심지역 매칭 | O |
| `RATE_DECISION` | 2(+서울전체) | 정책·금리 관심자에게 즉시 | O |
| `TRANSACTION_NEW` | 1 | **기본 요약**. 실거래 알림 옵트인 + 법정동 좁힘 + final ≥ 4일 때만 즉시 후보 | O |
| `MARKET_INDEX_UPDATED` | 2 | 기본 요약. 급변 시 즉시 | O |

### 6-4. D-day 정책

| 시점 | 발송 대상 | base 보정 |
| ---- | ---- | ---- |
| D-7 | 별 4↑ 공고 | +1 |
| D-3 | 관심지역/유형 매칭 | +2 |
| D-1 | 모든 매칭 | +3 |
| 당일 오전 | 모든 매칭 | +4 |
| 마감 후 | 발송 안 함 | - |

방해금지(`dnd_start`/`dnd_end`, Asia/Seoul)에 걸리면 즉시 발송하지 않고 다음 허용 시간에 큐잉. 단 당일 마감은 사용자가 허용한 경우 예외 발송.

### 6-5. 중복 방지

- `notification_dedup_key = anonymousId + event_type + ref_type + ref_id + scheduled_date`
- 같은 공고의 D-3/D-1/당일은 `scheduled_date`가 달라 별도 발송 가능.
- 알림 끄면 토큰 유지 + `push_enabled=false`. 발송 실패 3회↑ 토큰 비활성화.

### 6-6. outbox 소비 모델 (★ 추가 — 이벤트→알림 파이프라인)

`domain_event`를 트랜잭셔널 아웃박스로 쓰고, **Spring 알림 워커가 폴링**한다.

```
[Spring @Scheduled(every 30~60s)]  ← 단일/소수 워커
1) 미처리 이벤트 클레임 (동시성 안전)
   SELECT * FROM domain_event
   WHERE status = 'NEW' AND scheduled_at <= now()
   ORDER BY created_at
   LIMIT 100
   FOR UPDATE SKIP LOCKED;
   → UPDATE status='CLAIMED', claimed_at=now()

2) fan-out: 이벤트별 발송 대상 사용자 조회
   - 관심지역 매칭 + 관심유형 매칭 + 알림 강도 충족 사용자 select
   - 사용자별 personalized_delta 계산 → final_score → 발송/요약 판정

3) dedup: notification_logs(anonymousId, dedup_key) UNIQUE 위반이면 SKIP

4) 발송: FCM/WebPush. 성공/실패/요약(DIGEST)을 notification_logs에 기록

5) UPDATE domain_event status='DONE', processed_at=now()
   실패 시 retry_count++; 임계 초과 시 status='FAILED' + 내부 알림
```

요점: ① 사용자별 점수는 **여기(2단계)에서** 계산하지 이벤트에 저장하지 않는다. ② `SKIP LOCKED`로 워커 재시작/다중화에도 중복 처리 없음. ③ 발송 멱등성은 `notification_logs` UNIQUE가 최종 방어선. ④ 하루 요약은 `DAILY_DIGEST_ONLY`/요약 판정분을 모았다가 별도 스케줄(사용자 지정 시각, Asia/Seoul)에 1건으로 발송.

### 6-7. 비회원·푸시 보안

| 항목 | 정책 |
| ---- | ---- |
| anonymousId | UUID. 이름·전화·이메일 등 직접 식별정보 미저장 |
| 토큰 저장 | `user_devices`에 device token·권한상태·마지막 성공/실패시각 |
| 알림 해제 | `push_enabled=false`, 필요 시 토큰 삭제 |
| 데이터 삭제 | `anonymous_users.status=DELETED`, preferences/devices/logs 비활성화·삭제 |
| 기기 변경 | MVP 자동 복구 불가. 2차 로그인 시 병합 |

### 8-1. MVP (사용자 ~수천 명)

```
[ 단일 클라우드 VM / 소형 매니지드 — Docker Compose ]
 ├─ Spring API 서버      (서빙·비회원·알림 워커, 컨테이너)

 ├─ PostgreSQL          (공유 DB, 매니지드 권장 / SSOT)
 ├─ Flyway              (스키마 마이그레이션 — 단일 관리 지점)
 ├─ Redis               (Spring 캐시·간단 큐)
 └─ 오브젝트 스토리지       (raw_payload 아카이브, 선택)
```

- 두 컨테이너, 한 DB. Python 쓰기 / Spring 읽기.
- 스키마 변경은 Flyway로만. 쿠버네티스·Kafka는 과설계 → 확장 단계로.
- 시크릿(공공 API 키·FCM 키·LLM 키)은 환경변수/시크릿 매니저. 저장소 커밋 금지.

### 8-2. 모니터링

- **Prometheus + Grafana** 한 대시보드에 Spring API 지표(p95·에러율)와 Python 수집 잡 헬스(`collect_job_log` 기반 마지막 성공시각·건수·실패율)를 묶는다. "수집 실패 = 콘텐츠 공백"이라 수집 헬스가 1순위 알림.
- Spring: Actuator→Micrometer / Python: `prometheus_client`.

### 8-3. CI/CD

- GitHub Actions **모노레포 + 경로 필터**(`api/**`→Spring, `collector/**`→Python). 배포는 독립.
- 마이그레이션은 배포 파이프라인 단일 단계로 먼저 적용 후 앱 롤아웃.

### 8-4. 확장 (v1 이후)

수집 잡 증가→Airflow, 이벤트 버스→Kafka. 지도/히트맵→PostGIS+타일 캐시. 트래픽→API 수평 확장+읽기 캐시.

### interest_type (관심유형 — 목업 9종)

사용자 노출 라벨이자 알림/피드 필터. 각 유형은 아래 "매칭 대상"으로 이벤트와 연결된다.

| 코드 | 표시 | MVP 데이터 | 매칭 대상(supply_type / event / source) |
| ---- | ---- | ---- | ---- |
| `PRIVATE_SALE_SUB` | 민간청약 | O | supply_type {PRIVATE_SALE, OFFICETEL} |
| `UNRANKED_SUB` | 무순위 청약 | O | {UNRANKED} |
| `HAPPY_HOUSE` | 행복주택 | O | LH {HAPPY_HOUSE, NATIONAL_RENTAL} |
| `YOUTH_SAFE_HOUSE` | 청년안심주택 | **2차(SH)** | SH source — MVP 콘텐츠 없음, "개발 예정" 표시 |
| `LONG_TERM_JEONSE` | 장기전세 | **2차(SH)** | SH source — MVP 콘텐츠 없음, "개발 예정" 표시 |

## 12. 기획안 대비 보완 요약

1. **중요도 점수 분리** — 이벤트 base_score(Python 저장) ↔ 사용자별 personalized_delta(Spring 계산). domain_event엔 base만.
2. **outbox 소비 모델** — `status`/`claimed_at`/`SELECT FOR UPDATE SKIP LOCKED`로 중복·누락 없는 fan-out.
3. **신규등록 감지** — `first_seen_at` diff. 국토부 등록일 미제공 가정.
4. **집값전망 심리** — ECOS 서울 CSI 확정(`HOUSE_PRICE_OUTLOOK_SEOUL_CSI`).
5. **정책 일정** — MVP는 기준금리 결정일 seed만.
6. **실거래 알림** — 기본 푸시 OFF + 법정동 좁힘 옵트인.
7. **enum SSOT** — 부록 A 단일화.
8. **DDL/인덱스/유니크** — Flyway V1 초안(dedup_hash UNIQUE, 지역·월·first_seen 인덱스, outbox status 인덱스).
9. **에러 계약** — 공통 envelope + 코드표.
10. **PoC 0주차** — 5개 API 실호출 검증 + 호출량 산정 + fixture.
11. **시간 기준** — 전 계층 Asia/Seoul 고정.
12. **(v2.1) 목업 정합화** — 관심유형 9종 / 알림 강도 5종(`IMPORTANT_ONLY`=별 3개↑=final≥4) / 캘린더 단계 event_type 확장 / 집값전망은 서울 CSI 유지. 청년안심주택·장기전세·비기준금리 정책은 선택 가능하나 데이터 2차(개발 예정).

> 요약: v2.1의 MVP 백엔드는 "A등급 공식 API 배치 수집 + first_seen_at 변화감지 + base/개인화 분리 중요도 + outbox fan-out 알림 + 데이터 신뢰성 표시"로 구성되며, UI(관심유형/알림/캘린더)는 목업을 따르고, 0주차 PoC로 데이터 소스 가정을 먼저 검증한 뒤 1단계로 들어간다.
